# InterviewAce Project Report

Version: 1.0
Project Name: InterviewAce
Prepared on: 2026-04-07

## 1. Executive Summary
InterviewAce is an AI-assisted interview preparation platform that evaluates candidate answers using a hybrid scoring pipeline and provides personalized recommendations, session analytics, and research-grade reporting.

The system combines:
- AI concept evaluation
- Semantic similarity scoring using MiniLM
- Communication clarity scoring with guard rules

The final score is deterministic after component scores are computed, enabling repeatable reporting and clearer learning feedback.

## 2. A-to-Z Project Compendium

### A - Aim
Build a practical, measurable, and personalized interview coaching platform for technical and non-technical interview types.

### B - Business/Academic Problem Solved
Most interview tools either:
- focus only on coding tasks,
- provide inconsistent subjective feedback, or
- lack explainable, component-level scoring.

InterviewAce addresses this by combining model-based and deterministic scoring with transparent analytics.

### C - Core Capabilities
- Multi-type interview sessions: Technical, Behavioral, System Design, HR
- Role-aware question flow
- Hybrid scoring with structured feedback
- Session-level coach summary
- Personalized recommendations
- Analytics dashboard + research dashboard

### D - Data Model (High Level)
Main entities in MongoDB:
- Users / auth-linked identities
- Sessions
- Questions inside sessions
- Per-question evaluation metrics
- Coach summary
- User settings and profile context

### E - Evaluation Engine (Technical)
Current weighted fusion in code:
- Concept: 0.55
- Semantic: 0.30
- Clarity: 0.15

Final score:
S_final = 10 x (0.55 x C + 0.30 x S_sem + 0.15 x Cl)

Safety cap:
- If concept score <= 2, final score is capped at 40.

This is implemented in src/lib/evaluation/engine.ts.

### F - Feedback Style
Per-question output contains:
- explanation (interviewer-style)
- errors list
- strengths
- improvements

Session end includes aggregate coach summary with fallback if final AI call fails.

### G - Guardrails
AI evaluator is schema-constrained and validated to reduce malformed outputs and hallucination drift. Clarity guard logic applies word-count based bounds to prevent overly short responses from scoring unrealistically high.

### H - Hybrid Scoring Rationale
Each component captures different information:
- Concept score: correctness of reasoning
- Semantic score: closeness to reference meaning
- Clarity score: communication quality

Fusion improves robustness compared to any single channel.

### I - Interview Flow
Typical session stages:
1. Sign in
2. Configure role/type/difficulty/length
3. Ask question
4. Evaluate answer
5. Ask follow-up/next question
6. Generate coach summary and persist session

### J - JSON and API Discipline
AI responses are parsed and validated before scoring is finalized. Fallbacks are used when parsing or model calls fail.

Recommendation ranking is deterministic for identical profile/session state. It uses:
- skill deficits
- domain alignment
- difficulty fit
- onboarding weak areas
- trend-based adjustments (declining trend gets boost for foundational categories)

### S - Reported Research Result Snapshot
Based on the controlled dataset workflow and project manuscript baseline:
- Mean concept score: ~48.3
- Mean semantic score: ~52.5
- Mean hybrid score: ~54.8
- Correct class hybrid mean: ~71
- Partial class hybrid mean: ~60
- Incorrect class hybrid mean: ~34
- Correlation (concept vs semantic): ~0.020
- Standard deviation reduction vs concept-only baseline: ~37.4%

Note:
- Live dashboard values are computed from current DB data at runtime and can differ from this baseline snapshot.

## 4. Deployment Guide (Vercel)

### 4.1 Pre-Deploy Checklist
- Ensure lockfile is in sync with package.json
- Ensure environment variables are configured in Vercel
- Ensure GitHub OAuth callback URL matches deployed domain

### 4.2 Required Environment Variables (Production)
- NEXTAUTH_SECRET
- NEXTAUTH_URL=https://<your-vercel-domain>
- GITHUB_CLIENT_ID
- GITHUB_CLIENT_SECRET
- MONGODB_URI
- OPENROUTER_API_KEY (optional if fallback-only mode desired)
- OPENROUTER_MODEL

### 4.3 GitHub OAuth Configuration
OAuth app settings must include:
- Homepage URL: https://<your-vercel-domain>
- Authorization callback URL: https://<your-vercel-domain>/api/auth/callback/github

Mismatch here causes redirect_uri errors on login.

### 4.4 Vercel Build/Runtime Notes
Recent deployment fixes applied:
- lockfile mismatch resolved
- sharp runtime conflict mitigated by dependency/config updates
- pnpm build-script allowlist configured for required native/build packages

### 4.5 Deploy Steps
1. Push to master branch
2. If project already connected to Vercel, auto-deploy runs
3. If first deployment, import repo in Vercel dashboard and deploy manually once
4. After env var changes, trigger Redeploy

### 4.6 Post-Deploy Validation
- Login via GitHub
- Start and complete one interview session
- Verify session appears in dashboard
- Verify research page loads for admin
